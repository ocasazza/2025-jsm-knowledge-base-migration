Ubuntu 20+ general information:

  • LDAP credentials will grant you access to the workstation
  • Your $HOME directory is not shared, instead it's mounted on the NVMe drive for faster IO
  • Regional shares under /nfs are still available (except from VLAN51 (the 172.18.254.0/24 network) )
  • /scr is the mountpoint for the 1TB HDD (default hardware config)
  • /scr-sdd is just a directory on the NVMe with the same permissions as /scr , but may be used for fast version of scratch
  • By default, the assigned user may use sudo to reboot or poweroff their workstation
  • Updates ( apt and snap ) take place every morning (3AM local time) but the system is NOT set to automatically reboot if required to apply an update
    • Please take note to reboot the system every few weeks/months (you'll see a message upon an SSH login)

Developing on Linux

  • Linux machines can be managed or unmanaged.
  • If a Linux machine is managed, you get NFS access but no sudo access.
  • If a Linux machine is unmanaged, you get sudo access but no NFS access.
  • If you want to use Docker, you'll get a machine configured to use it. You can now use Docker whether or not you have a managed or an unmanaged machine.


How does an assigned user login to their workstation?

  • And because the assigned user has their home directory created at build time, the assigned user may login to the system from the GUI out-of-the-box.

How does an unassigned user login to a workstation?

  • In this case, the system hasn't created the user's home directory, but the home directory will get created upon a successful ssh connection to the desktop.

  • And because the non-assigned user doesn't have their home directory created at build time, the non-assigned user may not login to the system from the GUI out-of-the-box. The non-assigned user must first ssh into the system for the desktop's GUI to be accessible for them to login to the system's GUI.

How to troubleshoot SSH issues?

How to deal with REMOTE HOST IDENTIFICATION SSH warnings?

  • What to do if you see this warning message: 
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!     @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
IT IS POSSIBLE THAT SOMEONE IS DOING SOMETHING NASTY!
Someone could be eavesdropping on you right now (man-in-the-middle attack)!
It is also possible that a host key has just been changed.
  • This message means that the ssh host keys of the remote system have changed from the last time you connected to the remote system.
  • It likely means your system has been re-provisioned between now and the last time you attempted to connect to the system.


  • You may remove the old entry by deleting the line in question from your local "known_hosts" file, or you may remove the old entry just by running the command below:
    • ssh-keygen -f "/PATH/TO/YOUR/.ssh/known_hosts" -R "<REMOTE_HOST_NAME>"


  • But if you see this warning message and you know the system was not re-provisioned, please alert infosec as it might indicate that your system has been altered by someone other than Schrodinger IT.

How to switch desktop GUI environments on Ubuntu (Gnome, KDE, Cinnamon, or XFCE)?

  • By default, some systems might be set to use a desktop GUI that's not configured correctly at startup, or is having issues for an unknown reason.

  • Also, please note that Ubuntu 22.04 has some default options to use Wayland over X11/xorg
    • We recommend using a desktop environment that mentions "X11" or "xorg" to  ensure the desktop environment is not using Wayland
      • The following items are know issues on Wayland:
        • Druva's inSync Client
        • Appgate Client

  • No matter the reason, if you wish select another desktop GUI environment, such as Gnome, KDE, Cinnamon, or XFCE, then you may do so at the GUI login screen.

  • For the image examples below, " a" denotes Ubuntu 20.04 screenshot and " b" denotes Ubuntu 22.04 screenshot.


    • Please follow these directions to choose another desktop GUI environment:
      1. Make sure you're at the desktop's login screen:
      2. Select "Other" or "Not listed" from the user menu, then enter your Unix username or select your Unix username if it's already in the user list:
      3. After selecting or entering your Unix username, please enter your Okta password, but do NOT hit enter and do NOT click "Log In" just yet!
      4. Before clicking "Log In" or hitting the "Enter" key, go to the upper or lower right corner of the screen and click on the cogwheel (sometimes it might show up as a gnome-foot icon):
      5. After you've selected your desktop GUI environment, you may then click the "Log In" button, and you should then get logged into the desktop GUI you selected. If you didn't get to the correct desktop GUI environment, then you likely selected the GUI first before entering your user details (the GUI selector get reset once you type anything in the user or password fields).




How backups work on the Ubuntu workstations

  • The old CentOS7 desktops used shared NFS home directories (coming from their regional NFS server ([nyc|pdx]- stor.schrodinger.com ).
    • This means that the entire home directory (/home/<UNIX_USERNAME>) was backed up for a few weeks on the regional NFS server.


  • However, the new Ubuntu desktops do not have shared NFS home directories!
    • Your home directory on an Ubuntu desktop is locally stored on the M.2 SSD (which will make it much more responsive).
    • Ubuntu desktop users should rely on Druva's inSync client to backup any data they wish.
      • Please note, that Druva will only backup data under any of the following scratch directories:
        • /scr-ssd         (this is just a directory that's on the desktop's M.2/SSD; just faster scratch)
        • /scr-nvme     (points to an additional M.2 NMVe drive; this is NOT standard and most builds will not see this mountpoint)
        • /scr                 (typically, this is the desktop's 1TB HDD)
        • /scr2
        • /scr3
        • /scr4
        • /scr5
        • /scr6
        • /scr7
        • /scr8
      • If you have a directory under your home directory that you want backed up, some find it more convenient to first move the directory under /scr-ssd, and then create a symlink from the scratch directory to your home directory, such as: 
        • mv /home/<UNIX_USERNAME>/foo /scr-ssd/ && ln -s /scr-ssd/foo /home/<UNIX_USERNAME>/foo
        • mv /home/<UNIX_USERNAME>/bar /scr-ssd/ && ln -s /scr-ssd/bar /home/<UNIX_USERNAME>/bar

      • Here's an example of some useful symlinks to create: 

        • LIST="
          .config
          .local
          .mozilla
          Desktop
          Documents
          Downloads
          Pictures
          "
          for FOLDER in $LIST ; do
            [ -d /home/$USER/$FOLDER ] && mv /home/$USER/$FOLDER /scr-ssd/
            ln -s /scr-ssd/$FOLDER /home/$USER/$FOLDER
          done

      • We may include other directories, but for now, Druva will only backup data that is under any of the listed scratch directories.




How to access your old shared home directory from the Ubuntu desktop (not yet in place)

  • Although the new Ubuntu desktops don't have shared NFS mounted home directories, the old NFS home directory may temporarily be mounted read-only under /mnt/old-home-dirs on their new Ubuntu workstations
  • The /mnt/old-home-dirs mount-point will only be available for a few weeks .
  • The /mnt/old-home-dirs mount-point is only meant for users to migrate some of their old shared CentOS home directory data to their new non-shared home Ubuntu directory.



Troubleshooting Minor Issues:

How to deal with the bash timeout (TMOUT) issue?

  • By default, there is a bash terminal timeout which is set to 900 seconds in /etc/profile.d/tmout.sh ( TMOUT=900 )
  • Before disabling the default timeout, please consider using a program such as [tmux| https://wiki.archlinux.org/title/tmux ] or [screen| https://wiki.archlinux.org/title/GNU_Screen ] if you require a persistent terminal.

  • But if you must disable the TMOUT variable, you may add the following line to your local ~/.bashrc file: 
    • TMOUT=0
