Executive summary

On April 18th, 2022, at 8:14 AM EST Karthik Ganeshan reported that Opengrok was down. Yellaturu Kullay Babu followed up in the IT-SOS chat room asking for assistance with the issue. Opengrok was giving a Proxy error. (The Proxy server received an invalid response from the upstream server.) The proxy server could not handle the request GET /. There was an error reading from then remote server.

Background

OpenGrok is a source code search and cross reference engine so this service being down affects the work of developers across the organization.

Incident Description

Users attempting to access the OpenGrok webserver found it down and contacted the IT Support Hotline for assistance. We responded when notified and looked into the cause in order to restore the service.

Chronology

was notified at 12:28 AM on April 18th that pdx-opengrok-lv03 check failed via pagerduty.

The Alert resolved automatically through the integration API.

Karthik Ganeshan reported that Opengrok was down at 8:14 AM EST.

Babu followed up in the IT-SOS hotline at 8:56 AM.

responded in the IT-SOS chatroom and began to look into the issue at 9:34 AM.

Informed the IT Support Hotline  that we were looking into the issue.

responded that the issue was fixed at 9:54 AM.

Corrective Action

restarted the Apache HTTP Server on pdx-opengrok-lv03.

Analysis

After restarting the service the search field took a couple of minutes to work. I'm not sure if that's how openGrok works.

Detection

How quickly was the problem detected?

The outage was detected immediately by LibreNMS and Pagerduty, however we didn't respond until the reports by users.

How was the problem detected?

LibreNMS and Pagerduty.

How can detection be improved?

Re-adjusting team rules/assignments  in Pagerduty.

Response

How quickly did we respond once detected?

We responded as soon as the issue was acknowledged.

Did we adequately communicate to our users?

Yes we did we communicate in the IT support Hotline.

If not, how can this be improved?

We could have improved our user communication by responding immediately when the issue occurred.

We should standardize and document a procedure for incident response.

Mitigation

What went well that helped minimize the impact of this problem?

When this issue was acknowledged by the team that was on call we responded immediately. Trying to fix the issue and informed the users in the IT support-line that w were actively working on the issue

What can be done to minimize the impact if a problem like this recurs?

Better monitoring, cleaning up some rules in pagerduty so that this issue is noticed and visible to the team before our end-users notice. Maybe we should follow up on suggestion and review how we're assigning incidents by team in PagerDuty and, if necessary, get more explicit about which teams the after-hours on-call should be examining.

Avoidance

What can we do in the future to avoid a problem like this?

We should upgrade Open Grok as this issue has become a frequent occurrence. We should look more into the log and figure out why this happens in the first place. I think at the current moment we aren't really sure why the service keeps failing.
