Executive summary

On 19th December 2022, there was a power outage in Mannheim and as its result, the whole district was without electricity. Our uninterruptible power supplies provided electricity for the server room only.

Background

There are two APC Smart-UPS 3000 in the server room.

Incident Description


Chronology

Time (local time) Event

10:40

UPS: On battery power in response to distorted input.

10:43

The power outage has been reported on google chat (Mannheim Office)

10:47

UPS: The power for switched outlet group 1-3, Outlet Group 1-3, is now turned off.

10:48

Office manager confirmed the whole building is without power. The technicians were working on it.

10:50

Office manager informed that there was a massive power outage in Mannheim which caused the power outage in the building.

11:18

Office manager informed that the power outage was expected to be fixed by 13:00 local time.

11:50

UPS: A discharged battery condition no longer exists.

Corrective Action

The power outage in Mannheim was resolved and the electricity was restored.

Analysis

The time zone was incorrect on mhg-ups-02, which has been updated to CET (UTC+1). The runtime is not balanced. UPS-01: 6mins, UPS-02: 33mins at the current load.

Detection

How quickly was the problem detected?

The problem has been detected quickly.

How was the problem detected?

The problem was detected and reported by user and via email alert.

How can detection be improved?

Monitoring both UPSs via LibreNMS.

Response

How quickly did we respond once detected?

We respond relatively fast but there was no room for any further action because of the root of the problem.

Did we adequately communicate to our users?

The communication occurred via Google Chat.

If not, how can this be improved?

We could have informed all DE users in email about the incident.

Mitigation

What went well that helped minimize the impact of this problem?

The incident was detected fast and informed Mannheim users via Google Chat.

What can be done to minimize the impact if a problem like this recurs?

The running time can be increased and distribute the load better on the two UPSs.

Avoidance

What can we do in the future to avoid a problem like this?

It is very hard to avoid this type of issue because it occurred outside of the perimeter. We can expand the running time and put all desktops or some of them onto UPS.
