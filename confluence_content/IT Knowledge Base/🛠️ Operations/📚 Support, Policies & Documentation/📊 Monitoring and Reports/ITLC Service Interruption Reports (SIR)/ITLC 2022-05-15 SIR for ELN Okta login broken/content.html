Executive summary

On the application vendor, Scilligence upgraded our ELN software from v6.7 to v6.8. When they did this they neglected to copy/install newer versions of the Okta sign-in widget that was installed for v6.7 back in January to address System JIRA ac6def98-2140-30c3-8103-4b52b8b31135 SYSMGR-61606 . This caused our Okta based authentication for Schrodinger employees to fail. This failure went unnoticed or unreported until System JIRA ac6def98-2140-30c3-8103-4b52b8b31135 ITDW-326 was opened on and resolved on .

Background

We have adopted a shared responsibility model for this application with the vendor Scilligence. We've been responsible for the administration of the VM, operating system and installation and maintenance plans for the database and they in turn take responsibility for testing and upgrading new versions of the application software.

It is also worth noting here that the Okta authentication is only active an necessary for Schrodinger internal employees and direct contractors. Our CROs login via an application-based authentication so having Okta login breaks has zero consequence to the bulk of the ELN users.

Incident Description

Back in January working with the vendor Scilligence we resolved a similar authentication failure with Okta that was resolved by updating to a more current version of the Okta web sign-in widget that is used to provide centralized authentication for Schrodinger employees. When the vendor updated our software on they failed to carry these fixes forward and re-installed older versions that didn't work.

Chronology

- Scilligence installs v6.8 and fails to update to the latest Okta web sign-in widget

- Noticed the problem, initiated the associated outage ticket, opened support ticket with Scilligence. Response from Scilligence put me on the path of suspecting the tie back to SYSMGR-61606.

- Screen sharing troubleshooting session with the vendor to identify the problem.

Corrective Action

Files that were added to v6.7 in the D:\Scilligence\Schrodinger\App\ELN6.7.0f36940\Web\tools\okta were copied to the corresponding location under D:\Scilligence\Schrodinger\App\ELN6.8.0f41041 . Browser cache was flushed and application authentication was tested successfully.

Analysis

There wasn't much independent analysis needed here. The parallels between this incident and the prior one in January was highly suggestive and the simple test of copying files from the version that used to work to the one that didn't was a trivial test case.

Detection

How quickly was the problem detected?

This problem was not detected quickly. It was 2 months between the introduction of the problem and the resolution. For the bulk of that time we were unaware of the problem having no means of detecting it through automation nor any reports from end-users with problems.

How was the problem detected?

Issue was detected by IT staff while trying to execute an unrelated ELN task.

How can detection be improved?

I can think of two ways to improve this detection.

  1. Establish an automated test using pingdom to test web login. This approach has some potential drawbacks and costs to it.
    1. We'd have to establish a test account with minimal privileges that would just exercise the login function
    2. Establishing that account would consume a license and the licensing for this product is costly.
  2. Increase the visibility of this problem in our user community and educate them to report authentication problems to us. I think it's possible that other people within Schrodinger may have encountered this problem but without reporting it.

Response

How quickly did we respond once detected?

We responded with reasonable swiftness to the initial report from IT Operations within the first hour of my day starting.

Did we adequately communicate to our users?

Since we'd gone so long with this problem unidentified it's reasonable to presume that the only user affected here was the IT Ops personnel who reported it and communication there was carried out through direct IM and ticket.

If not, how can this be improved?

Communication seems adequate to the situation with no improvement needed.

Mitigation

What went well that helped minimize the impact of this problem?

The primary mitigator here is that, as noted in the background, our CRO users aren't dependent on this authentication mechanism working.

What can be done to minimize the impact if a problem like this recurs?

Better and faster detection would head off those hidden impacts (unreported problems) that seem likely to have occurred. Analysis of our Okta logs or application logs might uncover the extent to which this problem went unreported but I suspect the answer there isn't worth the investment necessary to get it.

Avoidance

What can we do in the future to avoid a problem like this?

Better discipline by our vendor and/or some increased involvement from us
