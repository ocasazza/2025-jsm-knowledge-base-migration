Executive summary

This SIR was filed several months after the incident due to being on leave.

NYC license server service was reported down or very slow by multiple users in the IT Support Hotline. Investigation on nyc-lic-lv01 found high load, due to / being 100% full.  Freeing space and restarting the license manager services fixed the problem.

Background

nyc-lic-lv01 is a virtual machine in NYC and is an internal-use FLEXlm license server for the Schrodinger Suite and some third-party software from Intel and TotalView.

Incident Description

NYC license server service was reported down by multiple users in the IT Support Hotline. Investigation on nyc-lic-lv01 found high load, due to / being 100% full.  Freeing space and restarting the license manager services fixed the problem.

Chronology

Sep 19, 12:54 PM report by Jeff Saunders in the IT Hotline

Sep 19, 3:30 PM Shane pinged in IT - SOS

Sep 19, 3:43 PM Jon started investigating

Sep 19, 4:10 PM Jon reported back to the IT Hotline the issue was resolved

Corrective Action

Freeing space in / by copying off and truncating large files in /var/log/flexlm/ elsewhere, and restarting the license manager services.

Analysis

When checking on nyc-lic-lv01 and checking routine status items, like load and local filesystem usage, Jon noticed the load was high and / was 100% full. Freeing space in / and restarting the license manager services fixed the problem.

On further investigation (months later, in Jan 2023), from the old schrod.log FLEXlm log which had been moved to nyc-nx-l01:/scr/jon/schrod.log, I found messages that seem to indicate at some point after Sep 6, 2022, the license server could intermittently not serve more licenses because it was out of file descriptors (below).  Due to the way dates are logged in the FLEXlm log, it is not clear what exact date after Sep 6 corresponds to these first messages about "This license server system can handle no more concurrent clients since it is out of file descriptors."

Detection

How quickly was the problem detected?

/ being full was not detected until this ticket was filed. Checking /var/log/messages*, it seems the disk full messages started as long ago as Jun 30 2022:

messages-20220829.gz:Jun 30 02:27:01 nyc-lic-lv01 rsyslogd: imjournal: fopen() failed for path: '/var/lib/rsyslog/imjournal.state.tmp': No space left on device [v8.24.0-57.el7_9 try http://www.rsyslog.com/e/2013 ]

nyc-lic-lv01 is not in LibreNMS currently; if it were, then maybe / filling up would have generated an alert?

However, / filling up doesn't seem to have immediately caused license checkout failures, because it doesn't seem likely end-users would not have noticed for several months until Sept, when this ticket was created.

Analysis of the old schrod.log flexLM log indicates there were intermittent problems serving licenses starting sometime after Sep 6, but it's not clear exactly when between Sep 6 and Sep 19 those started.  If we go by these messages from the old log (nyc-nx-l01:/scr/jon/schrod.log):

and the chat message from Jeff at 12:54, then it took about 3 hours for a user to detect and report the problem.
