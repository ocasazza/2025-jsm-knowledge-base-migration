NX gives you access to a graphical desktop on a Linux machine from a remote computer. The Linux machine to which you are connecting can be either your own Linux desktop machine in the office, or one of certain shared-use NX gateway Linux machines specifically set up for NX access.

IMPORTANT REMINDER

YOU MUST use the exact NoMachine 5.3.12 clients attached to the bottom of this page. DO NOT USE newer versions downloaded directly from the NoMachine website, they will not work.

User Guide

If you've already setup and configured your client, you can find additional information in our User Guide .

NX client program setup

You will need to install an NX client program in order to connect to the Linux machine via NX. You will also need to create configuration entries ("sessions") for each Linux machine to which you wish to connect.

Use the following table to help you determine how to configure your NX client software:


Mode Connecting to Connecting from a machine “Host” setting in NX client "Protocol", and/or “Port” setting in NX client NX key to use in client
A your Linux desktop on the office wired network hostname of your Linux desktop SSH, 22 /etc/nxserver/client.id_dsa.key or /usr/local/src/nxclient/ hostname of your Linux desktop .client.id_dsa.key from your Linux desktop
B your Linux desktop on the VPN (outside the office, or on the office wireless) hostname of your Linux desktop SSH, 22 /etc/nxserver/client.id_dsa.key or /usr/local/src/nxclient/ hostname of your Linux desktop .client.id_dsa.key from your Linux desktop
C your Linux desktop using an ssh tunnel (outside the office, or on the office wireless) localhost SSH, local forwarded port /etc/nxserver/client.id_dsa.key or /usr/local/src/nxclient/ hostname of your Linux desktop .client.id_dsa.key from your Linux desktop
D an NX gateway (nyc-nxgate or pdx-nxgate) anywhere hostname of the NX gateway SSH, 22 The key file in /home/share/nx/ which corresponds to the NX gateway machine to which you wish to connect. Note: You will not be able to ssh into the NX gateway machine to retrieve the key file. You'll need to ssh into one of the ssh gateways (ny-gate or pdx-gate) or an office Linux machine other than the NX gateway in order to retrieve the key file from /home/share/nx/ .


Retrieving key files

You'll need to use scp or sftp, from the command-line or via a graphical client (like WinSCP on Windows, or CyberDuck on the Mac) to retrieve the key files mentioned in the table above. For example, using Terminal on a Mac to copy the key file for your Linux desktop to your Mac:

sftp _hostname of your Linux desktop_
get /usr/local/src/nxclient/ hostname of your Linux desktop .client.id_dsa.key

Remote NX via ssh tunnel (VPN is not required)

In the examples below, 8822 is the local forwarded port as mentioned in the table above. You don't need to use 8822 as your local forwded port as in the examples below, but that is a port unlikely to already be in use on your client machine. If 8822 is already in use on your client machine, you can choose some other unused port.

  • Linux
    • ssh -NfL 8822:<ip address or host name of remote NX server>:22 <username>@[ny-gate|pdx-gate]. schrodinger.com
  • Windows
    • Install PuTTY for Windows
    • Create a script file, start_ssh_tunnel.bat, with following line, and save to desktop:
      • For 32-bit windows: 
        • "C:\Program Files\PuTTY\plink.exe" -L 8822:&lt;ip address or host name of remote NX server&gt;:22  &lt;username&gt;@[ny-gate|pdx-gate].schrodinger.com
      • For 64-bit windows: 
        • "C:\Program Files (x86)\PuTTY\plink.exe" -L 8822:&lt;ip address or host name of remote NX server&gt;:22  &lt;username&gt;@[ny-gate|pdx-gate].schrodinger.com
        • For NX servers on NYC network use ny-gate; NX servers on PDX network, use pdx-gate
        • The ip address of an NX server can be found on the NX server with: 
          •  /sbin/ifconfig eth0 | grep 'inet addr:' | cut -d: -f2 | awk '{ print $1}'
    • Double click on start_ssh_tunnel.bat to create a ssh tunnel. Log in with your Linux account and leave the window open.
    • Create a new session
      • Configure the session following the steps listed above for All Platforms except following:
        • Host: 127.0.0.1
        • Port: 8822

  2. See more recent tips in our NX User Guide and CentOS 7 Known Issues .
  3. If you're using the NoMachine 4.x client on Windows, are able to successfully pass the login portion of the connection to the NX server, but then get 'Error is 108: Connection reset by peer' when trying to start an actual GNOME, KDE, etc. session, trying deleting or moving aside your ~/.nx directory on the NX server.
  4. Test if you are able to ssh into the remote server from some other machine in that same office. If you are not able to ssh in by using the remote server's hostname, or are able to ssh in but get a message saying that your home directory could not be found, it could be the remote server is booted into the wrong VLAN or has gotten a DHCP address. It's also possible the sssd daemon has crashed or gotten into a bad state and needs to be restarted. Sysmgr will need to look into it further in any of these cases.
  5. You can test the NX key of the remote server by copying it to your local machine and running:


    ssh -i the client.id_dsa.key file -o PasswordAuthentication=no nx@the remote machine Or, on the remote server you can run:

    ssh -i /var/lib/nxserver/home/.ssh/client.id_dsa.key nx@localhost

    If the key is working you should get something like this: HELLO NXSERVER - Version 3.2.0-74-SVN OS (GPL, using backend: not detected)NX> 105 If it isn't working, you can ask a sysmgr to re-generate the NX key on the remote server (instructions below are for non-salted, CentOS 5 or 6 desktops):

  6. Sessions cleanup

    End user

    If connecting to an NX gateway, note the actual server in the NX pool mentioned in the Detail output. For instance, if you're trying to NX into pdx-nxgate, the Detail output will mention if your failed connection was actually to pdx-nx-l01 or pdx-nx-l02. ssh to that NX server (pdx-nx-l01 or pdx-nx-l02, for example) in the NX pool, or to whichever machine to which you're trying to NX (your Linux desktop, for example) and kill any nxagent and nxnode processes owned by you . Also, delete any X* files owned by you in /tmp/.X11-unix/ on the NX server.  Finally, you can optionally move aside your /home/<user>/.nx directory on the NX server if the previous didn't help, and repeat the previous steps.

    You may need to repeat the above process one or more times, because each time you NX into one of the gateway machines, you'll likely be sent to a different actual machine in the pool, each with it's own set of processes and session files.

  7. To turn off mouseover effects (they slow down refreshing of the system menu), click "Settings", then "Configure the Panel". In the "Layout" section, click on the "Appearance" tab, uncheck "Enable icon mouseover effects". Click "Apply".
  8. If using NX Client, click Detail in the failure dialog and look at the output to see if there's anything obvious mentioned.
  9. Kill any nxagent and nxnode processes owned by you on the machine to which you're try to connect. Also, delete any X* files owned by you in /tmp/.X11-unix/ on that remote machine. Finally, move aside your /home//.nx directory on that remote machine. Then try to NX in to that remote machine again.
  10. If you're able to connect, but see a black screen instead of a desktop session, try to see if the issue is KDE- or Gnome-specific. That is, if you configure your NX client to launch KDE or GNOME on the remote server, does one work but the other not? Daemons associated with one of these desktops may need to be restarted (might be simpler to reboot the remove machine, if possible).
  11. If your NX session to a linux server fails to connect, check your shell config files on the linux machine ( .bashrc or .cshrc ). If you are setting $LD_LIBRARY_PATH manually you need to comment that out as it causes NX on the remote end to fail in some cases.
  12. If you are having issues with Gnome applets failing to start, try removing some temporary files: rm -rf /tmp/{orbit,virtual,ksocket,mapping,gconfd}-${USER}*
  13. If you're having NX issues, try checking the box for 'Disable DirectDraw for screen rendering' under the 'Advanced' tab of NX client options. Some discussion here . This issue appears to only affect machines with either Windows Vista or Windows 7.
  14. DanS - On Linux, if you are having bad performance that is otherwise unexplainable, check out the server machine. Multiple "nxagent"'s running caused my performance to be bad. Occurred twice, not sure how. I killed all except the most recent "nxagent", and the performance greatly improved.
  15. If seen "Error is 22: Invalid argument", try making connections a few times. This issue is only seen with NoMachine 4.0.x. Try installing latest 4.x release.
  16. User can't start NX session on pdx-nx-l01. Errors report out of space, but there is plenty disk space. The cause is running out inode. Some user's Gnome programs create lots of ORBit socket files and leave them behind. These socket files end up using most of available inodes. Fix: set tmpwatch to clean up old ORBit socket files in /tmp. Details in SYSMGR-28716
  17. If you are receiving error 108 on a Windows machine and a previous version of NoMachine was installed on the system, it may help to delete the cache-* folders from %USERPROFILE%\.nx
