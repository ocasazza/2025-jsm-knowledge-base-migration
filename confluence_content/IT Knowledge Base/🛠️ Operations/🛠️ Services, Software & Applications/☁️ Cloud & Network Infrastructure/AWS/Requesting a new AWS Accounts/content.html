Introduction

In order to both standardize our AWS account configuration, apply baseline industry best practices, and improve our overall cloud governance and security footprint, some basic information is needed before we create a new AWS account.  This is not an approval process, but collecting the details needed so that the account can be created and delivered with minimal back-and-forth.

Questions, comments, feedback, concens, etc. can be sent to .

(IT, this is the public document, the engineering details are in Service Documentation under Amazon Web Services.)

Questions for Requestor

  1. What's a good name for the account?

    1. What is it going to be used for?  This helps us understand a broader context.

    2. We are going to be asked "Why not GCP?"  Please give us a sentence or two to help us respond to that question.

  2. What group is going to be responsible for whatever and everything that happens within this new account?

    1. One to three individuals who will be overall responsible for everything (Will be added to "Owners" contact for the account, which is a super-set that includes the next three, and can manage all four groups)

    2. One to three individuals to be added to the "Infosec" contact, will be notified of any security or abuse issues, and can be point contacts for any needed Infosec remediation efforts. (Owners get all Security emails)

    3. Individuals or a team email group to be added to the "Ops" contact, who will be notified of AWS driven maintenance activities, like degraded hardware and server reboots. (Owners get all Ops emails)

    4. Optionally, individuals to be added to the "Billing" contact, who will get copies of monthly invoices, but no other AWS emails.  (Owners get all Billing emails)

  3. What stage in the environment lifecycle will this account be used for?  (a suffix will be added to the end of the account alias)

    1. This is a somewhat abstract descriptive list, very few projects need all stages, some projects only need one or two.  The purpose is to provide some operational context for how the account will be used, and to configure names and integrations to the rest of Schrödinger appropriately.

      1. sandbox / foo-sbox

        1. An environment where anything can happen at any time, including deletion of any resource.

      2. dev / foo-dev

        1. An environment where incremental changes happen based on developing a new feature or specific change.

      3. testing/QA / foo-test

        1. An environment where a group of incremental changes are made as a single larger change set for testing of various kinds

      4. staging / foo-stg

        1. An environment where validated and verified changes are made as a single deployment activity

      5. production / foo-prod

        1. An environment where a validated and verified deployment activity is made as the actual release

  4. Initial users to be given ReadOnly access ( Okta-ReadOnly )?

    1. https://docs.aws.amazon.com/IAM/latest/UserGuide/access_policies_job-functions.html#awsmp_readonlyaccess

  5. Initial users to be given PowerUser access ( Okta-PowerUser )?

    1. https://docs.aws.amazon.com/IAM/latest/UserGuide/security-iam-awsmanpol.html#security-iam-awsmanpol-IAMReadOnlyAccess

    2. https://docs.aws.amazon.com/IAM/latest/UserGuide/access_policies_job-functions.html#jf_developer-power-user

  6. Initial users to be given SuperAdmin access ( Okta-SuperAdmin )?

    1. https://docs.aws.amazon.com/IAM/latest/UserGuide/access_policies_job-functions.html#jf_administrator

  7. Are any other IAM roles needed?

    1. Other IAM roles will need to be setup in Okta, which may require a separate ticket.

  8. What is the VPC design?  This directly impacts the IP address CIDR block assignment.  IP address allocation is basically impossible to re-do, and trying to speed through this step creates significant network technical debt.

    1. What region(s)?  Each region needs a separate VPC and therefore a unique CIDR block.

    2. Will there be both private and public subnets?  How big are they?

    3. Will multiple availability zones be used?  How many?

    4. What's the total CIDR size needed for each VPC?

What the Requestor will get

  1. Mailing lists, with the specified "Owners" (#3 above) as both members and group owners, which will allow them to add and remove list members as needed. Managed at https://groups.google.com/

    1. aws-${account_alias}-owner@schrodinger.com

      1. aws-owner-schrodinger-all@schrodinger.com will be a member and contains those outside the requestor's group who need visibility into all AWS accounts, e.g., leadership and IT Infra.

    2. aws-${account_alias}-billing@schrodinger.com

      1. aws-billing-schrodinger-all@schrodinger.com will be a member and contains those outside the requestor's group who need visibility into all AWS accounts, e.g., accounting@schrodinger.com

    3. aws-${account_alias}-security@schrodinger.com

      1. aws-security-schrodinger-all@schrodinger.com will be a member and contains those outside the requestor's group who need visibility into all AWS accounts, e.g., infosec-team@schrodinger.com

    4. aws-${account_alias}-ops@schrodinger.com

      1. aws-ops-schrodinger-all@schrodinger.com will be a member and contains those outside the requestor's group who need visibility into all AWS accounts, e.g., it-infra@schrodinger.com

  2. Okta SSO enabled for the three standard roles. Each role has a group defined and managed within Okta.  Where possible, the named "Owners" will be given delegated Okta admin privileges to add and remove members of those groups themselves.  Managed at https://schrodinger-admin.okta.com/admin/dashboard

    1. aws:${account_alias}:${account_id}:Okta-ReadOnly

    2. aws:${account_alias}:${account_id}:Okta-PowerUser

    3. aws:${account_alias}:${account_id}:Okta-SuperAdmin

  3. The account will already be integrated with Wiz, an Infosec tool to monitor cloud resources

  4. The default VPCs in all regions will be deleted.

  5. An allocated internal CIDR block for VPC creation, based on the requestor's stated VPC design.

    1. If asked, IT Infra can collaborate in VPC implementation

    2. If specified, network access will be available and working.  The exact form is tbd, but could include

      1. Layer 3 routing via IPsec tunnels between NYC & PDX to the VPC

      2. Appgate access, details tbd

        1. gateway directly in vpc?

        2. via NYC/PDX and tunnels

  6. The root account credentials are stored securely, and available to IT Infra, Infosec, and IT leadership on those very rare occasions when they might be needed.

    1. https://docs.aws.amazon.com/accounts/latest/reference/root-user-tasks.html
