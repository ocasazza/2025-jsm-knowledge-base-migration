Executive summary

Boltsub1 hang issue started in April 2023.  This ticket was opened after the issue returned in June 2023.


Boltsub1 experienced high load and became unresponsive and was rebooted to bring service back.  Since then, a high load would recur from time to time,  which would last about 15-30 minutes and went away eventually by itself.  Such occurrences no longer totally hung boltsub1, but it did caused slowness and shell terminal unresponsiveness for a few minutes.  There were a huge amount of "NFS server not responding" messages logged by boltsub1.

Sysmgr looked at all currently mounted NFS file systems.  We found the NFS shares mounted from a particular Qumulo node (node 6) tend to lock up more often than mounted from other nodes.  Reported this finding to Qumulo support and Qumulo support confirmed Qumulo node 6 under-performed nfs service comparing to other Qumulo nodes.  As a result Qumulo node 6 chassis was replaced as suggested by Qumulo.

In the process of replacing node 6, a rare software bug with Qumulo OS/firmware was triggered on node 4.  The impact was that the mounted NFS shares from Qumulo node 4 were not responsive.  Temporarily removed Qumulo node 4 from service by removing node 4's dns records to restore NFS service on boltsub1 and other nodes after manually un-mounting the shares.   Due to this bug, bolt nodes can mount an NFS share from Qumulo node 4, but Qumulo node 4 could not serve NFS requests. Sysmgr upgraded Qumulo firmware from 6.0.1.1 to 6.1.0.1 per Qumulo support recommendation.

Shane set up some nfsiostat monitoring which showed evidence of high retransmissions and issues, but wasn't conclusive nor did it give more direction to find the cause of the problem.

To monitor NFS lock issue constantly, sysmgr deployed a cron task ( chk_nfs_mount) to log NFS hangs by file systems on boltsub1.  Certain Qumulo nodes logged most of NFS hangs.  Sysmgr looked switch ports stats and identified a few loose cable connections and then replaced several network cables.  This helped reducing number of NFS hangs on boltsub1.

Next, sysmgrs set up a local ldap server on bolt head node. Then switched ldap server configured on Qumulo cluster from ldap2 to the bolt ldap server.

After all above changes, there are still many "NFS server not responding" logged on boltsub1.   Such warning messages in general corresponds to some NFS hangs.  Another observation is that the bonding interface for internal network (which has MTU size 9000) experiences small, but constant package dropping, while the bonding interface for external network (which has MTU size 1500 and both use same physical NICs) does not experience any packet drops. In a way, the packet drops can cause NFS slowness in some high I/O utilization cases. After ITHPC discussion, sysmgrs decided to change MTU from 9000 to 1500 on boltsub1's internal network interface to match that of the external interface.   The MTU size change had a huge impact on error "Server not responding" and totally eliminated the error since the change.  Later sysmgrs made the same MTU change on all bolt nodes and Qumulo nodes.

Event with error "Server not responding"  being gone,  some NFS hangs were still logged on boltsub1.

We also noticed a dramatic difference in performance on boltsub1 from other nodes.  Running 

time $SCHRODINGER/testapp -t 30 -HOST bolt_personal

on boltsub1 took 1.5 minutes wheras on boltsub2 it would take <10s to complete.

Given we have looked all possible aspects and verified hardware states, system settings were fine, we came to an conclusion that boltsub1 must have some hardware issue not detected/reported by BMC/OS.  The final solution to this ongoing issue is to switch its functions to boltsub2, which is already completed.

The takeways from this incident are:

  1. When the issue seems be difficult to track down and resolve, move the service away to another performing node
  2. When troubleshooting this issue, do a quick sanity checking first:
    1. Check node hardware states, any hardware alerts from BMC/IPMI
    2. Check node software configuration, any recent changes; any system errors; any local file system fill-up; networking setting
    3. Check switch stats for this node and Qumulo nodes
    4. Check which NFS file systems were locked/hung, trace back to which Qumulo node(s). Check related Qumulo nodes
  3. Collect additional data regarding NFS stats; NFS hangs to help troubleshooting
  4. Reach out to Qumulo support for in-depth investigation on Qumulo cluster nodes

Background

Boltsub1 is a submission node on bolt cluster that allows end users to log in and perform certain tasks.  This node is used by job server processes to upload user data, files; prepare user jobs etc.

Incident Description

Boltsub1 would experienced high load in such a way that the load would go up quickly to up to 150 within 10-15 minutes.  Most of times, the high load dropped by itself after 20 minutes or so.  At this time some NFS mounted filesystems were hung.

Chronology

This issue started in April 2023. It occurred once and then came back in June.

Corrective Action

Sysmgr did troubleshooting. Checking boltsub1 hardware, system configuration and switch configuration.  Replaced Qumulo node 6.  Upgraded Qumulo OS from 6.0.1.1 to 6.1.0.3; Set up a local LDAP server; Changed MTU size from 9000 to 1500 on all internal network interfaces of all bolt nodes; Switched boltsub1 function to boltsub2
